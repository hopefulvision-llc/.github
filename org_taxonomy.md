# HopefulVision Repository Taxonomy

This document provides a high-level map of the HopefulVision GitHub organization.

The HopefulVision ecosystem is not a single software project. It is a constellation of philosophical, civic, architectural, and experimental repositories that together explore consciousness-aware technology, ethical governance, living intelligence, and civilizational design.

This taxonomy exists to help contributors, collaborators, and newcomers understand how the repositories relate to one another.

---

## Core Org Principles

- **Clarity over sprawl**
- **Distinct boundaries over conceptual overlap**
- **Demonstration over abstraction**
- **Consciousness, dignity, and consent over extraction**
- **Systems that serve life, not the other way around**

Examples:

## Repository Status Vocabulary

To improve clarity, repositories may use one of the following status labels in their README files:

- **Foundational Canon**
  Core worldview or normative material that defines major parts of the ecosystem.

- **Working Canon**
  Important and active conceptual material that is still evolving.

- **Working Spec**
  System or governance design material intended to guide future implementation.

- **Prototype**
  A functional or partially functional implementation artifact.

- **Public Interface**
  A repo intended primarily for external presentation or onboarding.

- **Experimental Branch**
  A speculative, emergent, or exploratory branch not yet stabilized.

- **Archive**
  Historical material retained for continuity but not actively developed.

---

## Boundary Guidelines

To keep the organization legible, each repo should make four things explicit near the top of its README:

1. **What this repo is**
2. **Why it exists**
3. **What it is not**
4. **Which related repos it connects to**

This helps reduce conceptual overlap and improves navigation across the ecosystem.

## Suggested Contributor Workflow

When contributing to a repository in this organization:

1. Determine the repo’s **layer**
2. Determine the repo’s **status**
3. Clarify whether the work is:
   - canonical
   - architectural
   - experimental
   - public-facing
4. Link related repos explicitly rather than duplicating material across them
5. Prefer sharpening boundaries over creating new overlap
